## Issues Fixed

### 1. ✅ Host Player Name Bug

**Issue**: When creating a lobby, the host's player name was being set to the lobby name instead of their actual player name.

**Example**:
- User "Alice" creates lobby "My Cool Game"
- Expected: lobby_name = "Alice's Lobby", player_name = "Alice"
- Actual: lobby_name = "My Cool Game's Lobby", player_name = "My Cool Game" ❌

**Root Cause**:
- [GoFishGameService.ts:80](packages/frontend/src/services/GoFishGameService.ts#L80) passed the lobby name (user input) to `createLobby(name, maxPlayers)`
- effectstreamBridge used this as the `playerName` in the blockchain command
- State machine received: `playerName = "My Cool Game"` instead of `playerName = "Alice"`

**Fix**: [GoFishGameService.ts:81](packages/frontend/src/services/GoFishGameService.ts#L81)
```typescript
// Before
const result = await EffectstreamBridge.createLobby(name, maxPlayers);

// After - Pass player's name, not lobby name
const result = await EffectstreamBridge.createLobby(this.playerName, maxPlayers);
```

**Impact**: Host's player name now correctly shows their actual name (e.g., "Alice") instead of the lobby name.

### 2. ✅ Missing Circuit Error (hasDealt)

**Issue**: Query function calling non-existent circuit method
```
ERROR [MidnightQuery] queryHasDealt failed: TypeError: queryContract.impureCircuits.hasDealt is not a function
```

**Root Cause**:
- `hasDealt` circuit exists in GoFish.compact but wasn't exported in TypeScript wrapper
- `queryHasDealt()` was trying to call `queryContract.impureCircuits.hasDealt()`
- Only `getCardsDealt` was available in the generated TypeScript bindings

**Fix**: [midnight-query.ts:291-310](packages/client/node/src/midnight-query.ts#L291-L310)

```typescript
// Before
const result = queryContract.impureCircuits.hasDealt(queryContext, gameId, BigInt(playerId));
return result.result;

// After - Use getCardsDealt and check if > 0
const result = queryContract.impureCircuits.getCardsDealt(queryContext, gameId, BigInt(playerId));
return Number(result.result) > 0;  // If dealt > 0 cards, has dealt
```

**Impact**: Setup status checks now work correctly without errors.

### 3. ✅ Automatic Setup Error Handling

**Issue**: Setup would throw errors and stop when encountering "already applied" errors during race conditions.

**Root Cause**:
- When status check returned stale data, frontend would retry operations
- "Player has already applied their mask" error would cause setup to fail
- Setup wouldn't continue even though the operation was actually successful

**Fix**: [GameScreen.ts:291-298](packages/frontend/src/screens/GameScreen.ts#L291-L298)

```typescript
// Before
if (!maskResult.success) {
  if (maskResult.errorMessage?.includes('already applied')) {
    console.log('[GameScreen] Mask already applied (detected via error)');
  } else {
    throw new Error(`Apply mask failed: ${maskResult.errorMessage}`);
  }
}

// After - Don't throw on "already applied", just continue
if (!maskResult.success) {
  if (maskResult.errorMessage?.includes('already applied')) {
    console.log('[GameScreen] Mask already applied (detected via error) - continuing');
    // Don't throw error, just continue to next step
  } else {
    throw new Error(`Apply mask failed: ${maskResult.errorMessage}`);
  }
}
```

**Impact**: Automatic setup is more robust and handles race conditions gracefully.
